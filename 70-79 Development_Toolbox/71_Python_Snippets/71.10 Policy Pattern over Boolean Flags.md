# Policy Pattern over Boolean Flags

## Overview
Boolean flag parameters (`do_audit=True`, `require_mfa=False`) accumulate into tangled `if/else` branches that violate SRP and OCP. The **Policy pattern** replaces flag-driven branching with a list of small, composable functions (or objects) executed in order — each policy decides what to do for its own concern.

> [!INFO] Source
> Inspired by ArjanCodes — *"Don't Use Boolean Flags in Python, Use Policies Instead"* ([examples repo](https://github.com/ArjanCodes/examples/tree/main/2026/policy)).

---

## The Problem: Flag-Driven Branching

A single function gradually grows a forest of conditionals as new rules appear:

```python
def process_request(user: User, request: Request) -> Request:
    if not user.is_active:
        raise PermissionError("Inactive users cannot make requests")
    if request.action == "delete" and not user.has_mfa:
        raise PermissionError("MFA required for delete actions")
    if request.required_role and request.required_role not in user.roles:
        raise PermissionError(f"Missing role: {request.required_role}")
    if request.requires_audit:
        request.audit_log.append(f"{user.name} {request.action} {request.path}")
    request.access_granted = True
    return request
```

**Problems:** hard to test in isolation, can't reorder/disable rules, every new rule edits the same function (OCP violation).

---

## Solution 1: OOP Policies (Protocol)

Each rule becomes a class implementing a shared `Protocol`; a `CompositePolicy` runs them in order.

```python
class RequestPolicy(Protocol):
    def apply(self, user: User, request: Request) -> None: ...

class MfaRequiredPolicy:
    def apply(self, user: User, request: Request) -> None:
        if request.action == "delete" and not user.has_mfa:
            raise PermissionError("MFA required for delete actions")

class CompositePolicy:
    def __init__(self, policies: list[RequestPolicy]) -> None:
        self.policies = policies
    def apply(self, user: User, request: Request) -> None:
        for p in self.policies:
            p.apply(user, request)
```

Good when policies need state, dependencies, or rich configuration.

---

## Solution 2: Functional Policies + Registry (Recommended)

Treat policies as pure functions, register them by name, and load the active set from configuration.

```python
Policy = Callable[[User, Request], Request]

def active_user(user: User, request: Request) -> Request:
    if not user.is_active:
        raise PermissionError("Inactive users cannot make requests")
    return request

POLICY_REGISTRY: dict[str, Policy] = {
    "active_user": active_user,
    "mfa_required": mfa_required,
    "role_required": role_required,
    "audit": audit,
    "grant_access": grant_access,
}

def apply_policies(user: User, request: Request, policies: list[Policy]) -> Request:
    return reduce(lambda r, p: p(user, r), policies, request)
```

Pair with `pydantic-settings` so the enabled list lives in `settings.env`:

```python
class Settings(BaseSettings):
    enabled_policies: list[str] = Field(default_factory=lambda: ["active_user", "mfa_required"])
    model_config = SettingsConfigDict(env_file="settings.env", env_prefix="APP_")
```

> [!TIP] Use immutable requests
> Return a new `Request` via `dataclasses.replace(...)` instead of mutating in place — composes cleanly with `reduce` and avoids hidden side effects.

## Comparison

| Approach | Reorderable | Config-driven | Testable in isolation | Best for |
|---|---|---|---|---|
| Boolean flags | ❌ | ❌ | ❌ | Trivial scripts only |
| OOP Composite | ✅ | Manual | ✅ | Stateful / DI-heavy rules |
| Functional + Registry | ✅ | ✅ (env / config) | ✅ | Most cases — pure rules, easy to extend |

---

## When to Reach for Policies
- A function has **3+ boolean parameters** that gate behavior.
- Rules need to be **toggled per environment** (dev vs prod).
- Different callers need **different subsets** of the same checks.
- You keep editing one mega-function whenever a new rule appears.

> [!WARNING] Don't over-engineer
> For two flags in a small script, a couple of `if` statements are fine. Reach for policies once the branching grows or rules need to vary independently.
