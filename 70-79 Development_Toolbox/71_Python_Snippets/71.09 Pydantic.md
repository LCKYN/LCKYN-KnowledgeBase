# Pydantic

## Overview
**Pydantic** is Python's most widely used data validation library. It uses type annotations to define schemas and validates data at runtime — raising clear errors on invalid input rather than letting bad data silently propagate. Pydantic v2 (2023) rewrote the core in Rust, making it **5–50× faster** than v1.

> [!INFO] Core idea
> Define a class with type-annotated fields → Pydantic validates and coerces input automatically. No manual `if isinstance(...)` checks.

---

## Quickstart

```python
from pydantic import BaseModel, Field, field_validator
from datetime import date

class User(BaseModel):
    id: int
    name: str
    email: str
    birth_date: date | None = None
    role: str = "viewer"   # default value

# ✅ Valid — coerces types where safe
user = User(id="42", name="Alice", email="alice@example.com")
print(user.id)    # 42  (str → int coerced)
print(user.role)  # "viewer" (default applied)

# ❌ Invalid — raises ValidationError with clear message
user = User(id="not_a_number", name="Alice", email="alice@example.com")
# pydantic_core.ValidationError: 1 validation error for User
# id: Input should be a valid integer [type=int_parsing]
```

## Core Features

### Field Metadata & Constraints

```python
from pydantic import BaseModel, Field

class Product(BaseModel):
    name: str = Field(min_length=1, max_length=100, description="Product display name")
    price: float = Field(gt=0, description="Price in USD, must be positive")
    quantity: int = Field(ge=0, le=10_000, default=0)
    tags: list[str] = Field(default_factory=list, max_length=10)
```

`Field()` constraints:
- Strings: `min_length`, `max_length`, `pattern` (regex)
- Numbers: `gt`, `ge`, `lt`, `le`, `multiple_of`
- Collections: `min_length`, `max_length`

### Validators

```python
from pydantic import BaseModel, field_validator, model_validator

class Order(BaseModel):
    amount: float
    currency: str
    discount: float = 0.0

    @field_validator("currency")
    @classmethod
    def currency_must_be_uppercase(cls, v: str) -> str:
        return v.upper()   # coerce rather than reject

    @field_validator("discount")
    @classmethod
    def discount_in_range(cls, v: float) -> float:
        if not 0 <= v <= 1:
            raise ValueError("discount must be between 0 and 1")
        return v

    @model_validator(mode="after")
    def net_must_be_positive(self) -> "Order":
        if self.amount * (1 - self.discount) <= 0:
            raise ValueError("net amount after discount must be positive")
        return self
```

### Nested Models

```python
from pydantic import BaseModel

class Address(BaseModel):
    street: str
    city: str
    country: str = "TH"

class Customer(BaseModel):
    name: str
    address: Address          # nested model — validated recursively
    tags: list[str] = []

# Nested dicts are auto-coerced to Address
customer = Customer(
    name="Alice",
    address={"street": "123 Main St", "city": "Bangkok"},
)
print(customer.address.country)   # "TH"
```

### Serialization

```python
user = User(id=1, name="Alice", email="alice@example.com")

# → dict
user.model_dump()
# → dict excluding None fields
user.model_dump(exclude_none=True)
# → dict with specific fields only
user.model_dump(include={"id", "name"})

# → JSON string
user.model_dump_json()
# → JSON pretty
user.model_dump_json(indent=2)

# From dict
User.model_validate({"id": 1, "name": "Alice", "email": "a@b.com"})
# From JSON string
User.model_validate_json('{"id": 1, "name": "Alice", "email": "a@b.com"}')
```

### JSON Schema Generation

```python
import json

# Full JSON Schema for the model (OpenAPI compatible)
schema = User.model_json_schema()
print(json.dumps(schema, indent=2))
# {
#   "properties": {
#     "id": {"title": "Id", "type": "integer"},
#     "name": {"title": "Name", "type": "string"},
#     ...
#   }
# }
```

> [!TIP] This is why Pydantic is used for LLM tool schemas
> `model_json_schema()` produces exactly what OpenAI, Anthropic, and vLLM expect for structured output and function calling. `Field(description=...)` text ends up in the schema and is read by the LLM — see [[23.05 Database Schema Documentation]].

## Common Patterns

### Optional Fields vs Required

```python
from pydantic import BaseModel

class UpdateUserRequest(BaseModel):
    # Required — must be provided
    user_id: int

    # Optional with default None — field exists but value may be absent
    name: str | None = None
    email: str | None = None
    role: str | None = None

# Partial update pattern — only set fields that were provided
def update_user(req: UpdateUserRequest):
    updates = req.model_dump(exclude_none=True, exclude={"user_id"})
    # updates only contains fields the caller actually sent
    db.update("users", req.user_id, updates)
```

### Strict Mode (no coercion)

```python
from pydantic import BaseModel

class StrictModel(BaseModel):
    model_config = {"strict": True}   # no type coercion
    id: int

StrictModel(id="42")   # ❌ raises error — str not coerced to int in strict mode
StrictModel(id=42)     # ✅
```

### Config Options

```python
from pydantic import BaseModel, ConfigDict

class MyModel(BaseModel):
    model_config = ConfigDict(
        str_strip_whitespace=True,    # auto-strip leading/trailing spaces
        str_to_lower=False,
        frozen=True,                  # immutable instances (like dataclass frozen=True)
        populate_by_name=True,        # allow both alias and field name
        extra="forbid",               # raise error on unknown fields (good for APIs)
    )
```

### Aliases (API field name ≠ Python name)

```python
from pydantic import BaseModel, Field, AliasChoices

class ExternalEvent(BaseModel):
    user_id: int = Field(alias="userId")         # JSON uses camelCase
    event_type: str = Field(alias="eventType")
    created_at: str = Field(
        validation_alias=AliasChoices("created_at", "createdAt", "timestamp")
    )

# Accepts camelCase from external API
event = ExternalEvent.model_validate({"userId": 1, "eventType": "click", "createdAt": "2025-01-01"})
print(event.user_id)   # 1
```

### Computed Fields

```python
from pydantic import BaseModel, computed_field

class Rectangle(BaseModel):
    width: float
    height: float

    @computed_field
    @property
    def area(self) -> float:
        return self.width * self.height

r = Rectangle(width=3.0, height=4.0)
print(r.area)                # 12.0
print(r.model_dump())        # {'width': 3.0, 'height': 4.0, 'area': 12.0}
```

## Pydantic with FastAPI

FastAPI uses Pydantic for request/response validation and auto-generated OpenAPI docs:

```python
from fastapi import FastAPI
from pydantic import BaseModel, Field

app = FastAPI()

class CreateOrderRequest(BaseModel):
    product_id: int = Field(description="The product to order")
    quantity: int = Field(gt=0, le=100, description="Units to purchase")

class OrderResponse(BaseModel):
    order_id: str
    total_usd: float

@app.post("/orders", response_model=OrderResponse)
async def create_order(body: CreateOrderRequest) -> OrderResponse:
    # body is already validated — no manual checks needed
    return OrderResponse(order_id="ord_123", total_usd=body.quantity * 9.99)
```

## Pydantic for LLM Tool Calls

```python
from pydantic import BaseModel, Field
from openai import OpenAI
import json

class WebSearchInput(BaseModel):
    query: str = Field(description="Search query string")
    max_results: int = Field(default=5, ge=1, le=20, description="Number of results to return")

client = OpenAI()

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Search for Python Pydantic tutorials"}],
    tools=[{
        "type": "function",
        "function": {
            "name": "web_search",
            "description": "Search the web",
            "parameters": WebSearchInput.model_json_schema(),  # ← schema from Pydantic
        }
    }]
)

# Parse and validate the LLM's tool call arguments
args = WebSearchInput.model_validate_json(
    response.choices[0].message.tool_calls[0].function.arguments
)
```

## v1 → v2 Migration Cheatsheet

| v1 | v2 |
|---|---|
| `from pydantic import validator` | `from pydantic import field_validator` |
| `@validator("field")` | `@field_validator("field")` |
| `class Config: ...` | `model_config = ConfigDict(...)` |
| `.dict()` | `.model_dump()` |
| `.json()` | `.model_dump_json()` |
| `.parse_obj(data)` | `.model_validate(data)` |
| `.parse_raw(json_str)` | `.model_validate_json(json_str)` |
| `.schema()` | `.model_json_schema()` |
| `__fields__` | `model_fields` |

> [!WARNING] v1 and v2 are not fully compatible
> If you use libraries that still depend on pydantic v1 (e.g., old LangChain versions), you may need `pip install "pydantic<2"` or use pydantic's v1 compatibility layer: `from pydantic.v1 import BaseModel`.

## Performance

Pydantic v2 core is written in Rust (`pydantic-core`):

| Operation | v1 | v2 | Speedup |
|---|---|---|---|
| Model validation | baseline | **5–50×** | Rust core |
| JSON serialization | baseline | **~10×** | Rust serializer |
| Schema generation | baseline | **~2×** | — |

For high-throughput APIs, v2 performance is comparable to manual dict validation.
