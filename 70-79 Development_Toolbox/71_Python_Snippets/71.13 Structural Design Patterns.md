# Structural Design Patterns

## Overview
**Structural patterns** describe how to compose classes and objects into larger structures while keeping those structures flexible. Most of them are variations on one move: *put an object in front of another object* â€” to translate (Adapter), simplify (Facade), restrict/augment access (Proxy), add behavior (Decorator), or share state (Flyweight).

---

## Pattern Summary

| Pattern | Intent | One-liner |
|---|---|---|
| **Adapter** | Convert one interface to another | "Make it fit" |
| **Bridge** | Split abstraction from implementation so both vary | "Two hierarchies, not one" |
| **Composite** | Treat single objects and groups uniformly | "Tree of parts" |
| **Decorator** | Add behavior by wrapping, not subclassing | "Layers of wrapping paper" |
| **Facade** | One simple entry point over a complex subsystem | "Front desk" |
| **Flyweight** | Share immutable state across many objects | "Intern it" |
| **Proxy** | Stand-in that controls access to the real object | "Gatekeeper" |

---

## Adapter

- Wraps an object with an incompatible interface so clients can use it through the interface they expect.
- Classic use: integrating a third-party SDK behind your own `Protocol` so vendors can be swapped.

```python
class OpenWeatherAdapter:  # implements our WeatherSource protocol
    def __init__(self, client: OpenWeatherClient) -> None:
        self._client = client
    def current_temp(self, city: str) -> float:
        return self._client.fetch(city)["main"]["temp"]
```

> [!TIP] Adapt at the boundary
> Keep adapters at the edges of the system; core logic should only see your own interfaces. This is the heart of ports-and-adapters (hexagonal) architecture.

---

## Bridge

- Splits one exploding class hierarchy into **two independent ones** â€” abstraction and implementation â€” connected by composition.
- Symptom it fixes: class names like `PdfInvoiceRenderer`, `HtmlInvoiceRenderer`, `PdfReportRenderer`, `HtmlReportRenderer` (NĂ—M explosion). Bridge: `Invoice(renderer)`, `Report(renderer)`.

---

## Composite

- Objects arranged in a **tree**; leaves and containers share one interface, so clients call `render()` / `price()` / `size()` without caring which they hold.
- Natural fit for file systems, UI widget trees, org charts, expression trees.

```python
class Node(Protocol):
    def size(self) -> int: ...

class File:      # leaf
    def size(self) -> int: return self._bytes

class Folder:    # composite â€” same interface, recurses
    def size(self) -> int: return sum(c.size() for c in self.children)
```

---

## Decorator

- Wraps an object with the **same interface**, adding behavior before/after delegating â€” stackable at runtime, unlike subclassing.
- GoF Decorator wraps *objects*; Python's `@decorator` wraps *functions* â€” same spirit (transparent wrapping), different mechanics.

> [!EXAMPLE] Stacking
> `CachedSource(RetryingSource(HttpSource()))` â€” caching and retry added independently, in any order, without a `CachedRetryingHttpSource` class.

---

## Facade

- A single, simplified interface over a messy subsystem; clients call `report_service.generate(month)` instead of orchestrating five modules themselves.
- Doesn't hide the subsystem â€” power users can still go around it.
- Cheap, high-value pattern: most "service" classes in applications are facades.

---

## Flyweight

- When millions of objects share identical immutable data, store that data **once** and reference it.
- Python built-ins already do this: string/small-int interning, `sys.intern`. Manual version: a cache dict keyed by the shared value.
- Rarely needed in application code; relevant for game entities, glyph rendering, large NLP token stores.

---

## Proxy

- Same interface as the real object, but the proxy **controls access**: lazy loading (virtual proxy), permissions (protection proxy), remote calls (remote proxy), caching.
- Contrast with Decorator: Decorator *adds behavior*; Proxy *manages access* â€” structure is identical, intent differs.
- See [[31.03 Lazy Loading]] for the virtual-proxy idea at system scale.

## Choosing Between the Wrappers

| You need toâ€¦ | Use |
|---|---|
| Change the interface | Adapter |
| Keep interface, add behavior | Decorator |
| Keep interface, control access | Proxy |
| Shrink the interface of many objects | Facade |
