# Database Schema Documentation

## Overview
**Schema documentation** means adding descriptions to tables, columns, views, and relations — through SQL `COMMENT` statements, dbt YAML, ORM docstrings, or metadata catalogs. The question is: **are they necessary?**

> [!INFO] Short answer
> For personal projects or scripts: optional. For team production systems, text-to-SQL tools, AI assistants, or data catalogs: **yes, necessary** — they are machine-readable context that directly affects query accuracy and maintainability.

---

## Why Descriptions Matter

### 1. Without Comments — What AI and Humans Must Guess

```sql
-- What is this table? What does "status" mean? What values are valid?
SELECT *
FROM ord
WHERE status = 2
  AND tp = 'B'
  AND amt_net > 0;
```

Column names `status`, `tp`, `amt_net` are ambiguous. Both a human and an LLM writing SQL must guess.

### 2. With Comments — Intent Is Explicit

```sql
-- Table: orders (all customer purchase orders)
-- status: 1=pending, 2=confirmed, 3=shipped, 4=cancelled
-- tp: order type — B=bulk, R=retail, W=wholesale
-- amt_net: net order amount in USD after discounts, before tax
SELECT *
FROM orders
WHERE status = 2      -- confirmed
  AND tp = 'B'         -- bulk orders only
  AND amt_net > 0;
```

## Impact on AI Tools

### Text-to-SQL (LLM-generated SQL)

Text-to-SQL models (GPT-4, Claude, Gemini, Defog SQLcoder) receive your schema as context. Column descriptions directly determine SQL correctness.

| Schema Quality | Text-to-SQL Accuracy | Example failure without comments |
|---|---|---|
| Column names only | ~60–70% on complex queries | `status = 1` vs `status = 2` — model guesses enum values |
| Names + types | ~70–80% | Can't resolve `tp` without domain knowledge |
| Names + types + descriptions | **85–95%** | Model knows exact enum values, units, join keys |

```
Prompt to LLM:
  "Get all confirmed bulk orders placed this month"

Schema without comments:
  orders(id, status, tp, amt_net, created_at)
  → LLM guesses: status='confirmed', tp=?   ← likely wrong

Schema with comments:
  orders(
    id         -- primary key
    status     -- 1=pending,2=confirmed,3=shipped,4=cancelled
    tp         -- B=bulk, R=retail, W=wholesale
    amt_net    -- net order value USD
    created_at -- UTC timestamp of order placement
  )
  → LLM generates: WHERE status=2 AND tp='B'   ← correct
```

### Copilot / AI Code Suggestions for ORM queries

When writing SQLAlchemy, Django ORM, or Prisma queries, Copilot reads your model definitions. Comments on fields give it the same information:

```python
# ❌ Copilot can't tell what these fields mean
class Order(Base):
    status: int
    tp: str
    amt_net: float

# ✅ Copilot generates correct filter conditions
class Order(Base):
    status: int     # 1=pending, 2=confirmed, 3=shipped, 4=cancelled
    tp: str         # order type: B=bulk, R=retail, W=wholesale
    amt_net: float  # net order value in USD, post-discount pre-tax
```

### Data Catalog / Semantic Search

Tools like **DataHub**, **Amundsen**, **Atlan**, **dbt Docs** index column descriptions for semantic search. Without descriptions, `SELECT * FROM ord WHERE tp='B'` is undiscoverable via search for "bulk orders".

## What to Document

### Always Document ✅

| Thing | Why |
|---|---|
| **Enum / status columns** | Numeric or single-char codes are meaningless without the mapping |
| **Abbreviations** (`amt`, `qty`, `tp`, `cust`) | Non-obvious meaning; can conflict across tables |
| **Business-defined terms** | "active user", "net revenue" — definitions vary by company |
| **Units and currency** | `amount` — is it cents or dollars? USD or THB? |
| **Nullable reasoning** | Why is this `NULL`? Is it unknown, not applicable, or a bug? |
| **Foreign key relationships** | Especially if not enforced by a FK constraint |
| **Soft-delete columns** | `deleted_at`, `is_active` — what does NULL mean? |
| **Timestamps — timezone** | UTC? Local? User's timezone? |

### Usually Sufficient Without Verbose Comments 📝

| Thing | Note |
|---|---|
| `id` / `uuid` | Obvious primary key — a one-liner suffices |
| `created_at` / `updated_at` | Obvious if timezone is documented at table level |
| Self-evident columns | `first_name`, `email` of a `users` table |

### Avoid ❌

| Thing | Why |
|---|---|
| Commenting every obvious column | `-- user's first name` on `first_name` is noise |
| Duplicating the column type | `-- integer field` adds nothing |
| Stale comments that contradict code | Worse than no comment — actively misleading |

## How to Add Descriptions

### PostgreSQL — Native `COMMENT ON`

```sql
COMMENT ON TABLE orders IS
  'All customer purchase orders. One row per order. Source of truth for revenue reporting.';

COMMENT ON COLUMN orders.status IS
  'Order lifecycle state: 1=pending, 2=confirmed, 3=shipped, 4=cancelled, 5=refunded';

COMMENT ON COLUMN orders.tp IS
  'Order type: B=bulk (business), R=retail (individual), W=wholesale (distributor)';

COMMENT ON COLUMN orders.amt_net IS
  'Net order value in USD. Post-discount, pre-tax. Negative values indicate credit orders.';
```

```sql
-- Query comments
SELECT
    col_description(attrelid, attnum) AS description,
    attname AS column_name
FROM pg_attribute
JOIN pg_class ON pg_class.oid = attrelid
WHERE relname = 'orders' AND attnum > 0;
```

### dbt — YAML Schema Documentation (recommended for analytics)

```yaml
# models/schema.yml
models:
  - name: orders
    description: >
      All customer purchase orders. One row per order.
      Source of truth for revenue reporting.
    columns:
      - name: status
        description: "Order lifecycle state: 1=pending, 2=confirmed, 3=shipped, 4=cancelled"
        tests:
          - accepted_values:
              values: [1, 2, 3, 4]

      - name: tp
        description: "Order type: B=bulk, R=retail, W=wholesale"

      - name: amt_net
        description: "Net order value USD. Post-discount, pre-tax."
```

> [!TIP] dbt is the gold standard for analytics schema docs
> dbt auto-generates a browsable data catalog (`dbt docs generate && dbt docs serve`) from YAML descriptions. Descriptions become searchable, lineage-linked, and version-controlled alongside the SQL models.

### SQLAlchemy ORM

```python
from sqlalchemy import Column, Integer, String, Numeric, DateTime
from sqlalchemy.orm import DeclarativeBase

class Base(DeclarativeBase):
    pass

class Order(Base):
    """All customer purchase orders. Source of truth for revenue reporting."""
    __tablename__ = "orders"

    id = Column(Integer, primary_key=True)
    status = Column(
        Integer,
        comment="Order lifecycle: 1=pending, 2=confirmed, 3=shipped, 4=cancelled",
    )
    tp = Column(
        String(1),
        comment="Order type: B=bulk, R=retail, W=wholesale",
    )
    amt_net = Column(
        Numeric(12, 2),
        comment="Net order value USD. Post-discount, pre-tax.",
    )
```

SQLAlchemy `comment=` is written to the database DDL when using `Base.metadata.create_all()`.

### Pydantic (for API schemas / tool schemas fed to LLMs)

```python
from pydantic import BaseModel, Field

class Order(BaseModel):
    status: int = Field(
        description="Order lifecycle: 1=pending, 2=confirmed, 3=shipped, 4=cancelled"
    )
    tp: str = Field(
        description="Order type: B=bulk (business), R=retail, W=wholesale"
    )
    amt_net: float = Field(
        description="Net order value in USD. Post-discount, pre-tax. Negative=credit."
    )
```

`Field(description=...)` is included in `model.model_json_schema()` — this schema is what gets sent to the LLM for function calling / tool use. Descriptions directly affect whether the LLM fills the fields correctly.

## Comment Quality Checklist

For each column that needs a comment, a good description answers:

- [ ] **What is it?** — business meaning, not technical type
- [ ] **What are the valid values?** — enum mapping, units, range
- [ ] **What does NULL mean?** — if nullable
- [ ] **What is the grain?** — one row per what?
- [ ] **Any caveats?** — timezone, currency, known data quality issues

---

## Summary

| Scenario | Descriptions Necessary? |
|---|---|
| Personal script, throwaway DB | ❌ Skip |
| Team production database | ✅ Yes — at minimum enums, abbreviations, units |
| Text-to-SQL / AI query generation | ✅ Critical — directly determines accuracy |
| Pydantic / tool schemas for LLM | ✅ Critical — LLM reads `Field(description=...)` |
| dbt analytics models | ✅ Yes — enables data catalog and governance |
| ORM models reviewed by Copilot | ✅ Recommended — `comment=` improves suggestions |
| Public API documentation | ✅ Yes — OpenAPI spec includes descriptions |
