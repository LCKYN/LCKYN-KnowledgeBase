# LLM Observability & Tracing

## Overview
**LLM observability** is the practice of capturing what an LLM application actually did in production — every prompt, completion, retrieval, and tool call — as structured **traces**. Unlike classic APM, the interesting failures aren't errors or latency spikes: a trace can be 200 OK, fast, and *wrong*. Tracing is the substrate that makes [[24.05 LLM-as-Judge]] scoring and [[24.06 Online Evaluation & Prompt Regression]] possible.

## What to Capture

A **trace** is one user request; **spans** are its steps:

- **Generation spans** — model, full prompt (with template version), completion, token counts, cost, latency (TTFT + total), finish reason
- **Retrieval spans** — query, retrieved chunks + scores (to debug [[11.12 RAG]] quality: was the answer wrong because retrieval missed?)
- **Tool spans** — [[11.03 LLM Tool Calls]] name, arguments, result, errors
- **Agent loops** — nested spans per step, so you can see *where* a multi-step run went sideways
- **Context** — user/session ID, prompt version, model version, experiment flag
- **Feedback** — thumbs up/down, edits, retries linked back to the trace

> [!TIP] Trace-first, metrics-second
> Aggregate metrics (avg latency, cost/day) tell you *that* something changed; only traces tell you *why*. Instrument traces from day one — you cannot retroactively capture the prompt that produced last week's bad answer.

## OpenTelemetry GenAI Conventions

[[24.03 OpenTelemetry]] now defines **GenAI semantic conventions** — standard span attributes like `gen_ai.request.model`, `gen_ai.usage.input_tokens`, `gen_ai.operation.name` — so LLM spans flow through your existing [[24.02 Observability Stack]] and vendor lock-in stays low. Most LLM observability tools can ingest or emit OTel.

## Tooling Landscape

| | **Langfuse** | **LangSmith** | **OTel + own stack** |
|---|---|---|---|
| Source | Open source, self-hostable | Proprietary (LangChain) | Fully yours |
| Strengths | Prompt management, evals, cost tracking | Deep LangChain/LangGraph integration, datasets | One stack for app + LLM |
| Fit | Default OSS choice, framework-agnostic | Teams already on LangChain | Strong platform teams |

Others: Braintrust, Arize Phoenix, W&B Weave, Helicone (proxy-based).

## What to Alert On

- **Cost & tokens** — per user/feature; runaway agent loops show up here first
- **Latency** — TTFT p95 for streaming UX
- **Error & refusal rates** — spikes often mean a provider or prompt regression
- **Feedback rate shifts** — thumbs-down trending up is your drift signal (LLM analog of [[24.01 Model Drift Detection]])
- **Judge-score drops** — sampled online scoring, see [[24.05 LLM-as-Judge]]

> [!WARNING] Traces contain user data
> Prompts and completions are PII-dense. Apply retention limits, masking/redaction at ingestion, and access control — the trace store is now one of your most sensitive databases.
