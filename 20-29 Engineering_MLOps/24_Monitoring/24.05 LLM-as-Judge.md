# LLM-as-Judge

## Overview
**LLM-as-judge** uses a (usually stronger) LLM to score another model's outputs against a rubric — quality, faithfulness, helpfulness, safety. It's the only evaluation method that scales to open-ended generation: exact-match metrics can't grade "write a good summary," and humans can't grade 50k traces a week. The catch: the judge is itself a model, with biases that must be measured, not assumed away.

## Judging Modes

| Mode | How | Best for |
|---|---|---|
| **Pointwise (direct scoring)** | Score one output on a 1–5 rubric | Monitoring absolute quality over time |
| **Pairwise comparison** | "Which of A/B is better?" | Model/prompt selection; more reliable than absolute scores |
| **Reference-based** | Compare output to a gold answer | Datasets with known-good answers |
| **Criteria decomposition** | Separate judgments per dimension (faithful? concise? safe?) | Actionable diagnostics; avoids one mushy score |

## Writing a Judge Prompt

- **One criterion per judge** — "rate overall quality" produces noise; "does the answer contain claims unsupported by the context? yes/no" produces signal
- **Binary or few-point scales** — yes/no or 1–3 beats 1–10 (models can't use fine-grained scales consistently)
- **Rubric with anchored examples** — show what a 1 and a 3 look like
- **Require reasoning before verdict** — [[11.13 Chain of Thought]] then a structured verdict field
- **Structured output** — force JSON so scores parse reliably

## Known Judge Biases

> [!WARNING] Measure these before trusting any judge
> - **Position bias** — in pairwise mode, favors the first (sometimes second) option → run both orders, keep only consistent verdicts
> - **Verbosity bias** — longer answers score higher regardless of content
> - **Self-preference** — models rate their own family's outputs higher → judge with a different model family than the one being judged
> - **Sycophancy toward confidence** — assertive wrong answers beat hedged right ones
> - **Rubric drift** — judge model updates silently shift scores → pin judge model versions like dependencies

## Calibration Against Humans

A judge is only usable once validated:

1. Human-label a sample (100–500 items) with the same rubric
2. Measure judge–human agreement (Cohen's kappa / correlation) — and human–human agreement as the ceiling
3. Iterate the judge prompt until agreement approaches that ceiling
4. Re-calibrate on a fresh sample periodically and on every judge-model change

## Practical Use Cases

- **Online monitoring** — sample production traces from [[24.04 LLM Observability & Tracing]], score asynchronously, alert on trends
- **Regression gates** — judge scores as pass/fail in CI ([[24.06 Online Evaluation & Prompt Regression]])
- **RAG faithfulness** — judge checks answers against retrieved context ([[11.12 RAG]] hallucination detection)
- **Data filtering** — same mechanics as [[11.50 Reward Models]] used offline
