# 11.35 Mixed Precision Training

## Overview
**Mixed precision training** uses lower-precision floats (FP16, BF16, FP8) for most operations while keeping a high-precision (FP32) **master copy of weights** for stable updates. It cuts memory ~2× and accelerates training 2–8× on modern GPUs/TPUs with negligible quality loss.

## Precision Formats

| Format | Bits | Range | Mantissa | Best For |
| --- | --- | --- | --- | --- |
| FP32 | 32 | ~$10^{\pm 38}$ | 23 | Master weights, optimizer state |
| **FP16** | 16 | ~$10^{\pm 4}$ | 10 | Activations on V100/T4 (needs loss scaling) |
| **BF16** | 16 | ~$10^{\pm 38}$ | 7 | Default on A100/H100/TPU; no loss scaling |
| **FP8** (E4M3 / E5M2) | 8 | varies | 3 / 2 | Forward / backward on H100, B200 |
| **TF32** | 19 (eff.) | ~$10^{\pm 38}$ | 10 | Auto on Ampere+ matmuls |

## Key Concepts
- **Master weights in FP32**: Updates accumulate without underflow
- **Compute in FP16/BF16**: Matmuls and activations use Tensor Cores (2–8× speedup)
- **Loss scaling** (FP16 only): Multiply loss by ~$2^{15}$ before backward to prevent gradient underflow; unscale before optimizer step
- **BF16 advantage**: Same dynamic range as FP32 → **no loss scaling needed**, much more stable
- **FP8 mixed recipes**: E4M3 for forward, E5M2 for backward (more range for gradients)

## FP16 vs BF16
> [!TIP] Use BF16 by Default
> On A100/H100/MI300/TPU, BF16 is strictly easier to use than FP16:
> - No loss scaling, no NaN handling, no overflow
> - Same throughput as FP16 on Tensor Cores
> - Slightly less precision (7 vs 10 mantissa bits) — rarely matters for LLMs

FP16 only makes sense on **older hardware** (V100, T4, consumer pre-Ampere) that lacks BF16 Tensor Cores.

## FP8 Training
- Hardware: **H100, H200, B200** (Hopper/Blackwell) Tensor Cores
- Frameworks: **Transformer Engine** (NVIDIA), **MS-AMP**, native PyTorch `torch.float8_e4m3fn`
- Requires **per-tensor or per-block scaling** to keep values in the narrow FP8 range
- Yields ~2× speedup over BF16 with comparable convergence on LLM pre-training

## Memory Math (7B Model, Adam Optimizer)

| Component | FP32 only | Mixed (BF16 + FP32 master) |
| --- | --- | --- |
| Weights | 28 GB | 14 GB (BF16) + 28 GB (FP32 master) |
| Gradients | 28 GB | 14 GB (BF16) |
| Adam states (m, v) | 56 GB | 56 GB (FP32) |
| **Total** | **~112 GB** | **~112 GB** |

> [!WARNING] Mixed Precision Doesn't Save Optimizer Memory
> Optimizer states stay in FP32 for stability — that's why **[[11.37 Distributed Training|ZeRO/FSDP]]** sharding matters even with mixed precision. Activation memory **is** halved, which is the main runtime saving.

## Practical Use Cases
- Default for all modern LLM training (pre-training, CPT, SFT)
- Combines with [[11.36 Gradient Checkpointing]] for further memory reduction
- Required to hit advertised TFLOPS on Tensor Core hardware
- FP8 increasingly standard for frontier-scale pre-training

> [!WARNING] Common Pitfalls
> - **FP16 NaN spikes**: Forgetting loss scaling or using too-high LR
> - **BF16 precision loss in normalization**: LayerNorm in BF16 can drift; keep in FP32
> - **Mixing precisions in residual streams**: Casting at every layer hurts; use autocast contexts
> - **FP8 calibration**: Wrong scaling factors cause silent quality degradation
