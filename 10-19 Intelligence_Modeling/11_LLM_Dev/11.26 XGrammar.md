# XGrammar

## Overview
**XGrammar** (by MLC AI) is a structured generation library that constrains LLM token sampling using **context-free grammars (CFG)**. It enables **zero-overhead** structured output — JSON, regex, custom grammars — without slowing down the LLM's decode throughput. XGrammar 2 (2025) introduced persistent execution engine, cross-request cache, and near-zero overhead on modern hardware.

> [!INFO] The problem it solves
> LLMs generate tokens probabilistically. Without constraints, they can produce invalid JSON, wrong field names, or hallucinated enum values. XGrammar masks invalid tokens at each step so the LLM **can only produce structurally valid output**.

---

## How It Works

### Token Masking via CFG

At each decode step, XGrammar computes a **bitmask** over the vocabulary:
- `1` = token is valid given the current grammar state
- `0` = token would violate the grammar

The LLM's logits are multiplied by this mask before sampling — invalid tokens get `-inf` probability.

```
LLM logits (all vocab):  [0.3, 0.1, 0.4, 0.2, ...]
Grammar mask:            [  1,   0,   1,   0, ...]
Masked logits:           [0.3, -inf, 0.4, -inf, ...]
                                        ↓
                              sample only from valid tokens
```

### XGrammar 2 Architecture

```
Grammar (JSON schema / EBNF / regex)
           │
           ▼
    Grammar Compiler
    (offline, one-time)
           │
           ▼
  Adaptive Token Mask Cache
  (pre-computes masks for common states)
           │
           ▼
  Persistent GrammarMatcher
  (tracks state across a full decode sequence)
           │
           ▼
  Per-step bitmask → applied to LLM logits → constrained sampling
```

**Key improvement in v2**: The mask cache is **persistent across requests**. For a given JSON schema used across 1000 requests, the grammar is compiled once and masks are cached — subsequent requests pay near-zero cost.

## Grammar Types Supported

| Type | Example Use Case |
|---|---|
| **JSON Schema** | Enforce API response structure, tool call outputs |
| **EBNF grammar** | Custom DSLs, SQL, code, markup |
| **Regex** | Phone numbers, dates, constrained string fields |
| **Built-in JSON** | Generic valid JSON (any structure) |
| **Structural tags** | `<json>...</json>` in thinking model outputs |

## Performance

XGrammar 2 benchmark vs alternatives (on H100, Llama-3.1-8B, complex JSON schema):

| Library | Overhead vs unconstrained | Notes |
|---|---|---|
| **XGrammar 2** | **< 1%** | Near-zero; mask computed concurrently with LLM prefill |
| **Outlines** | ~5–15% | CFG-based; recomputes per request |
| **Guidance** | ~10–30% | Token healing + CFG; higher overhead |
| **LMQL** | Higher | Query language approach |

> [!TIP] Why XGrammar is fast
> The vocabulary size is typically fixed (32K–128K tokens). XGrammar pre-computes which token transitions are valid for each grammar state offline. At runtime, applying the mask is just a bitwise AND — O(vocab_size / 64) with 64-bit words.

## Integration

### vLLM (native integration)
vLLM uses XGrammar as its default structured output backend:

```python
from vllm import LLM, SamplingParams
from pydantic import BaseModel

class CalendarEvent(BaseModel):
    name: str
    date: str
    participants: list[str]

llm = LLM(model="meta-llama/Llama-3.1-8B-Instruct")

params = SamplingParams(
    guided_decoding={"json": CalendarEvent.model_json_schema()},
    temperature=0.7,
)
output = llm.generate("Extract: Alice and Bob meet on March 5th", params)
```

### SGLang (native integration)

```python
import sglang as sgl
from pydantic import BaseModel

class Person(BaseModel):
    name: str
    age: int

@sgl.function
def extract_person(s, text):
    s += text
    s += sgl.gen("result", guided_json=Person)

state = extract_person.run(text="John is 30 years old")
person = Person.model_validate_json(state["result"])
```

### Standalone

```python
import xgrammar as xgr
from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-3.1-8B-Instruct")
compiler = xgr.GrammarCompiler(xgr.TokenizerInfo.from_huggingface(tokenizer))

# Compile once, reuse across requests
grammar = compiler.compile_json_schema(CalendarEvent.model_json_schema())
matcher = xgr.GrammarMatcher(grammar)

# Per decode step
token_bitmask = xgr.allocate_token_bitmask(1, tokenizer.vocab_size)
matcher.fill_next_token_bitmask(token_bitmask)   # apply to logits
# ... after sampling:
matcher.accept_token(sampled_token_id)
```

## XGrammar vs Alternatives

| Library | Approach | Overhead | Ease of Use | Best For |
|---|---|---|---|---|
| **XGrammar 2** | CFG + precomputed cache | Near-zero | Medium | Production serving, high throughput |
| **Outlines** | CFG per request | Low–medium | High | Ease of use, research |
| **Instructor** | Retry-based (no masking) | Low (extra LLM calls on failure) | Very high | Simple schemas, API use |
| **Guidance** | Token healing + CFG | Medium | Medium | Complex grammars |
| **llama.cpp grammar** | GBNF grammar | Low | Medium | Local/edge inference |

> [!INFO] Instructor vs XGrammar
> **Instructor** doesn't mask tokens — it asks the LLM to produce JSON and retries on parse failure. Simpler but can fail/retry. **XGrammar** guarantees valid output in one pass, no retries needed. Use Instructor for simple structured outputs via API; XGrammar for serving infrastructure.

## Structural Tag Support (Thinking Models)

XGrammar 2 adds support for models like **DeepSeek-R1** and **QwQ** that emit `<think>...</think>` blocks before the JSON answer:

```
<think>
Let me analyze this...
</think>
{"name": "Alice", "date": "2025-03-05"}
```

XGrammar can switch grammar mid-sequence: free text inside `<think>`, strict JSON after `</think>`.
