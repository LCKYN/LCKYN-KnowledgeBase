# 11.36 Gradient Checkpointing

## Overview
**Gradient checkpointing** (a.k.a. **activation checkpointing** or **rematerialization**) reduces training memory by **discarding intermediate activations during the forward pass and recomputing them during backward**. It trades ~30% extra compute for ~5–10× lower activation memory, enabling larger batches or longer sequences on the same hardware.

## The Memory Problem
During backprop, gradients require activations from the forward pass. For a transformer with $L$ layers, sequence length $s$, hidden size $d$, and batch $b$:

$$\text{Activation memory} \propto L \cdot s \cdot d \cdot b$$

For long sequences (e.g., $s = 32k$), activations dominate VRAM — often more than weights, gradients, and optimizer states **combined**.

## How It Works
- **Forward pass**: Save only activations at **checkpoint boundaries** (e.g., every transformer layer); discard the rest
- **Backward pass**: For each segment, **recompute** the discarded activations from the saved checkpoint, then compute gradients
- **Net effect**: Memory drops from $O(L)$ to $O(\sqrt{L})$ with optimal placement; compute increases by ~33% (one extra forward)

## Trade-off Summary

| Strategy | Activation Memory | Compute Overhead |
| --- | --- | --- |
| No checkpointing | $O(L)$ — full | 0% (baseline) |
| Checkpoint every layer | $O(1)$ per segment | ~33% (one extra forward) |
| Checkpoint every $\sqrt{L}$ layers | $O(\sqrt{L})$ — optimal | ~33% |
| Selective checkpointing | Tunable | <33% |

## Practical Use Cases
- **Long context training**: 32k–128k sequences without OOM
- **Larger batch sizes**: Better gradient quality and GPU utilization
- **Single-GPU fine-tuning**: Fitting 7B+ models with full FT on consumer cards
- **Combined with [[11.32 PEFT|QLoRA]]**: Standard recipe for fine-tuning 70B on a single 80GB GPU

## Selective / Smart Checkpointing
> [!TIP] Don't Checkpoint Cheap Ops
> Recomputing **attention** (especially with [[11.38 Flash Attention]]) is expensive; recomputing **LayerNorm + linear projections** is cheap. Modern frameworks (PyTorch `selective_checkpoint`, NVIDIA Transformer Engine) let you checkpoint only memory-heavy ops.

- **Full checkpointing**: Simplest, ~33% overhead
- **Selective**: Skip attention recomputation; ~10–15% overhead
- **Activation offloading to CPU**: Even less GPU memory, much higher latency — usually not worth it

## Framework Support
- **PyTorch**: `torch.utils.checkpoint.checkpoint` and `checkpoint_sequential`
- **HuggingFace Transformers**: `model.gradient_checkpointing_enable()`
- **DeepSpeed**: Activation checkpointing + partitioning across GPUs
- **FSDP**: Integrates with checkpointing for combined memory savings

> [!WARNING] Common Pitfalls
> - **Dropout / RNG mismatch**: Recomputed forward must use the **same random state**; use `use_reentrant=False` (PyTorch) or seeded RNG
> - **Incompatible with `torch.compile`** in some configurations — check before enabling
> - **Slower than expected**: If activations weren't the bottleneck, checkpointing only adds compute with no memory win
> - **Don't checkpoint LoRA-only training**: Adapters are tiny; activation memory dominates and full checkpointing helps, but the absolute savings are smaller than for full FT

## Memory Example (7B, seq=8k, batch=4, BF16)
| Config | Activation Memory |
| --- | --- |
| No checkpointing | ~48 GB |
| Full checkpointing | ~6 GB |
| Selective (skip Flash Attn recompute) | ~10 GB |

Combined with [[11.35 Mixed Precision Training]] and [[11.32 PEFT]], gradient checkpointing makes single-GPU LLM fine-tuning practical.
