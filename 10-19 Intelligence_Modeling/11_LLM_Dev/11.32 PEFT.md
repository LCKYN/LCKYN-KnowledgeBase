# 11.32 PEFT

## Overview
**Parameter-Efficient Fine-Tuning (PEFT)** adapts large pre-trained models by training only a **small fraction (typically <1%)** of parameters — either added modules or low-rank deltas — while keeping base weights frozen. It dramatically reduces VRAM, storage, and training cost with minimal quality loss.

## Key Concepts
- **Frozen base**: Pre-trained weights $W_0$ stay fixed; only small trainable parameters $\Delta\theta$ are updated
- **Composability**: Multiple task-specific adapters can be swapped on the same base model at inference
- **Storage**: Adapter checkpoints are MBs instead of GBs (a 7B model is ~14GB; a LoRA adapter is ~20MB)
- **Quality**: For most SFT tasks, PEFT matches full fine-tuning within 1–2% on benchmarks

## PEFT Methods Compared

| Method | What's Trained | Trainable % | Inference Overhead | Notes |
| --- | --- | --- | --- | --- |
| **LoRA** | Low-rank matrices $A, B$ s.t. $\Delta W = BA$ | ~0.1–1% | None (mergeable) | Default choice |
| **QLoRA** | LoRA on top of 4-bit quantized base | ~0.1–1% | Slight (dequant) | Fits 65B on single 48GB GPU |
| **DoRA** | Decomposes $W$ into magnitude + direction | ~0.1–1% | None (mergeable) | Closer to full FT than LoRA |
| **Adapters** | Small bottleneck MLPs inserted in layers | ~1–3% | Small latency add | Original Houlsby/Pfeiffer |
| **Prefix / P-Tuning** | Trainable virtual tokens prepended to keys/values | <0.1% | Adds prefix length | Good for prompt-style tasks |
| **IA³** | Learned scaling vectors on activations | <0.01% | Negligible | Extremely lightweight |

## LoRA Deep Dive
> [!INFO] Core Idea
> Weight updates during fine-tuning have **low intrinsic rank**. Approximate $\Delta W \in \mathbb{R}^{d \times k}$ as $BA$ where $B \in \mathbb{R}^{d \times r}$, $A \in \mathbb{R}^{r \times k}$, with $r \ll \min(d, k)$.

- **Effective weight at inference**: $W = W_0 + \frac{\alpha}{r} BA$ — can be **merged** into $W_0$ for zero overhead
- **Typical hyperparameters**: $r \in \{8, 16, 32, 64\}$, $\alpha = 2r$, dropout 0.05–0.1
- **Target modules**: Apply to attention projections (`q_proj`, `k_proj`, `v_proj`, `o_proj`) and often MLP (`gate_proj`, `up_proj`, `down_proj`)
- **Higher $r$**: More capacity, more VRAM; diminishing returns past $r=64$ for most tasks

## QLoRA
- Base model loaded in **NF4 (4-bit normal float)** quantization
- LoRA adapters trained in **bf16** on top
- **Double quantization** + **paged optimizers** for memory spikes
- Enables fine-tuning a **65B model on a single 48GB GPU**, or 7B on consumer 16GB

## Practical Use Cases
- Many task-specific specializations from one base (one base + N adapters)
- Personal/on-device fine-tuning (consumer GPUs)
- Rapid iteration during experimentation
- Multi-tenant serving with [[11.27 LLM Generation Engines]] supporting adapter hot-swapping (vLLM, SGLang)
- Domain adaptation when full fine-tuning is too expensive

> [!TIP] When to Use Full FT vs PEFT
> - **PEFT**: Default for SFT, style transfer, format learning, single-domain adaptation
> - **Full FT**: Major distribution shifts, continued pre-training, when you have abundant compute and want maximum quality

> [!WARNING] Common Pitfalls
> - **Wrong target modules**: Attention-only LoRA underperforms; include MLP layers for harder tasks
> - **Rank too low**: $r=4$ may bottleneck complex tasks
> - **Merging + quantization conflict**: Merged LoRA into a quantized base loses precision — keep separate or merge in fp16 first
> - **Learning rate**: PEFT tolerates **10–100× higher LR** than full FT (e.g., 2e-4 vs 2e-5)
