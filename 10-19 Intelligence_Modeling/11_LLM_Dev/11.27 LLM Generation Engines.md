# LLM Generation Engines

## Overview
An **LLM generation engine** (inference serving engine) is the software layer that takes a model's weights and efficiently serves token generation requests. The engine handles: KV cache management, batching, quantization, tensor parallelism, and structured output — so you don't have to implement these from scratch.

Choosing the right engine determines your **throughput, latency, hardware requirements, and operational complexity**.

---

## Do Engines Actually Improve Performance?

**Yes — dramatically.** The naive baseline (`model.generate()` in HuggingFace Transformers) is functionally correct but leaves most GPU performance on the table.

### Naive vs Engine: Numbers

| Metric | HuggingFace `generate()` | vLLM / SGLang | Difference |
|---|---|---|---|
| Throughput (Llama-3 8B, A100) | ~200–400 tokens/s | **1500–3000 tokens/s** | **4–8×** |
| GPU memory utilization | 60–70% (fragmented KV) | 85–95% | +25–35% |
| Concurrent requests | 1–4 (OOM beyond) | 50–200+ | **10–50×** |
| Time to first token (batch) | High (waits for full batch) | Low (continuous batching) | **2–5×** |

> [!INFO] Why the naive baseline is so slow
> HuggingFace `generate()` was designed for **research correctness**, not production throughput. It allocates KV cache statically for the max sequence length, processes one batch at a time, and doesn't overlap compute with memory transfers.

### What Each Optimization Buys

| Technique | What it fixes | Speedup |
|---|---|---|
| **Continuous batching** | Naive: waits until all requests in batch finish. Engine: adds new requests mid-batch | 2–5× throughput |
| **PagedAttention** | Naive: KV cache pre-allocated at max length → 60–80% wasted. Engine: paged → ~5% waste | 2–4× more concurrent users |
| **FlashAttention kernel** | Naive: attention reads/writes HBM repeatedly. FA2: fused tiled kernel, stays in SRAM | 1.5–3× for long contexts |
| **Quantization (FP8/INT8)** | Naive: FP16. Engine: smaller weights → fewer bytes loaded per token | 1.5–4× decode speed |
| **Speculative decoding** | Naive: 1 token/forward pass. Engine: draft proposes N, target verifies in one pass | 1.5–3× latency |
| **Tensor parallelism** | Naive: single GPU. Engine: splits weight matrices across GPUs | Near-linear scaling to 8 GPUs |
| **Prefix caching** (RadixAttention) | Naive: recomputes shared prefix every request. Engine: cache hit → skip prefill | 2–10× for shared-prompt workloads |
| **CUDA graph capture** | Naive: Python overhead per step. Engine: captures decode loop as static graph | 10–20% decode speedup |

### Efficiency Gains Summary

```
Same hardware, same model:

  Naive (HF generate):
    GPU: ████░░░░░░ 40% utilization
    KV:  ████████░░ 80% wasted (static alloc)
    Batch: 4 requests at a time

  vLLM:
    GPU: █████████░ 90% utilization
    KV:  █░░░░░░░░░ 5% waste (paged)
    Batch: 100+ requests concurrently

  Cost implication: serve 10–50× more users on the same hardware
```

> [!TIP] When does it matter most?
> - **High concurrency** (many users) → continuous batching + PagedAttention
> - **Long context** (RAG, agents, documents) → FlashAttention + KV quantization
> - **Repeated system prompts** (chatbots, multi-tenant) → prefix caching / RadixAttention
> - **Latency-sensitive** (interactive chat) → speculative decoding + CUDA graphs

## vLLM

**vLLM** (UC Berkeley, 2023) is the de facto standard open-source engine. Its core innovation is **PagedAttention** — managing KV cache memory in non-contiguous pages like OS virtual memory, eliminating fragmentation.

### Key Features
- **PagedAttention** — near-zero KV cache memory waste; enables large batch sizes
- **Continuous batching** — new requests join mid-batch without waiting
- **Speculative decoding** — built-in support (draft model + prompt lookup)
- **XGrammar** — default structured output backend (see [[11.26 XGrammar]])
- **Tensor + pipeline parallelism** — multi-GPU out of the box
- **Model support** — Llama, Mistral, Qwen, DeepSeek, Gemma, Phi, etc.

```python
from vllm import LLM, SamplingParams

llm = LLM(
    model="meta-llama/Llama-3.1-8B-Instruct",
    tensor_parallel_size=2,    # 2 GPUs
    quantization="fp8",        # FP8 weight quantization
    kv_cache_dtype="fp8",      # FP8 KV cache
    gpu_memory_utilization=0.9,
)

params = SamplingParams(temperature=0.7, max_tokens=512)
outputs = llm.generate(["Explain RAG in one paragraph"], params)
```

```bash
# OpenAI-compatible server
vllm serve meta-llama/Llama-3.1-8B-Instruct \
  --tensor-parallel-size 2 \
  --quantization fp8 \
  --port 8000
```

## SGLang

**SGLang** (Stanford + UC Berkeley, 2024) is optimized for **multi-turn, agentic, and structured generation** workloads. Its core innovation is **RadixAttention** — a trie-based prefix cache that aggressively reuses KV cache across requests with shared prefixes.

### Key Features
- **RadixAttention** — automatic prefix cache sharing vs vLLM's manual prefix caching
- **Multi-call programs** — batch multiple LLM calls in one program (fork/join/loop)
- **XGrammar** — native structured output integration
- **Speculative decoding** — eagle, draft model support
- **Better for**: chatbots with shared system prompts, RAG (shared context prefix), agents that make many small calls

```python
import sglang as sgl

@sgl.function
def classify_and_respond(s, text: str):
    s += sgl.system("You are a helpful assistant.")
    s += sgl.user(text)
    s += sgl.assistant(
        sgl.gen("category", choices=["question", "complaint", "compliment"]) +
        "\n" +
        sgl.gen("response", max_tokens=200)
    )

# Batch 100 requests efficiently
states = classify_and_respond.run_batch(
    [{"text": t} for t in texts],
    num_threads=32,
)
```

### vLLM vs SGLang

| Workload | Prefer |
|---|---|
| Simple chat completions, high throughput | vLLM |
| Agentic workflows with many small calls | SGLang |
| Shared system prompt across many users | SGLang (RadixAttention) |
| Structured JSON generation | Both (both use XGrammar) |
| Max GPU utilization, NVIDIA hardware | TensorRT-LLM |

## TensorRT-LLM

NVIDIA's inference engine. Compiles models to **TensorRT** optimized graphs with NVIDIA-specific fused kernels.

### Key Features
- **FP8 native** — first-class FP8 on Hopper (H100) and Ada GPUs
- **In-flight batching** — equivalent to continuous batching
- **Tensor + pipeline parallelism** — optimized for NVLink multi-GPU
- **Highest throughput on NVIDIA hardware** — 20–30% faster than vLLM on H100 benchmarks

```bash
# Build engine (one-time, ~10 min)
trtllm-build \
  --checkpoint_dir ./llama-3.1-8b-hf \
  --output_dir ./llama-3.1-8b-trt \
  --gemm_plugin float16 \
  --max_batch_size 64

# Run inference
python run.py \
  --engine_dir ./llama-3.1-8b-trt \
  --max_output_len 512
```

> [!WARNING] High setup cost
> TensorRT-LLM requires engine compilation per model per GPU type. A compiled engine for H100 doesn't run on A100. Factor in build time and storage for multiple compiled engines.

## TGI (Text Generation Inference)

HuggingFace's production serving engine. Best for teams already in the HF ecosystem.

```bash
docker run --gpus all \
  -p 8080:80 \
  -v $PWD/data:/data \
  ghcr.io/huggingface/text-generation-inference:latest \
  --model-id meta-llama/Llama-3.1-8B-Instruct \
  --quantize bitsandbytes \
  --max-input-length 4096
```

- OpenAI-compatible API out of the box
- Built-in Prometheus metrics
- Flash Attention 2, PagedAttention, speculative decoding

## llama.cpp / Ollama

For **local and edge deployment** — runs on CPU, Apple Silicon (Metal), and NVIDIA GPUs.

### llama.cpp
- **GGUF format** — quantized model files (Q4_K_M, Q8_0, etc.)
- Runs Llama 3 70B at ~5 tokens/s on M2 Max (no GPU needed)

```bash
./llama-server \
  --model models/llama-3.1-8b-q4_k_m.gguf \
  --ctx-size 8192 \
  --n-gpu-layers 40 \  # offload 40 layers to GPU
  --port 8080
```

### Ollama
User-friendly wrapper around llama.cpp + additional backends:

```bash
ollama pull llama3.1          # download model
ollama run llama3.1           # interactive chat
ollama serve                  # start REST API at localhost:11434
```

```python
import ollama

response = ollama.chat(
    model="llama3.1",
    messages=[{"role": "user", "content": "Explain transformers"}],
)
```

## Choosing an Engine

```
Where are you deploying?
    │
    ├─ Cloud / data center (NVIDIA GPUs)?
    │     ├─ Max throughput, production → TensorRT-LLM
    │     ├─ Good throughput, easy ops → vLLM
    │     └─ Agentic / multi-call workloads → SGLang
    │
    ├─ HuggingFace ecosystem / quick start?
    │     └─ TGI
    │
    ├─ Local dev / laptop / no GPU?
    │     ├─ Technical setup → llama.cpp
    │     └─ Easy setup → Ollama
    │
    └─ Mobile / browser / multi-platform?
          └─ MLC-LLM
```
