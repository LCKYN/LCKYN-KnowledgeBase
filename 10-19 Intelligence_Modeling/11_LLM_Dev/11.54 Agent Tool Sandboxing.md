# Agent Tool Sandboxing

## Overview
**Agent tool sandboxing** is the practice of bounding what an [[11.02 LLM Agents|LLM agent]] can *do* — regardless of what it *decides*. Because [[11.53 Prompt Injection]] can hijack any model that reads untrusted content, the security boundary must live in the **tools and their permissions**, never in the model's judgment. Design assumption: the model is a confused deputy that will eventually follow attacker instructions.

> [!TIP] The one rule
> An agent's blast radius = the union of what its tools can reach. Secure the tools like you'd secure an intern's credentials, and the model's mistakes become recoverable instead of catastrophic.

## Least-Privilege Tool Design

- **Narrow tools over general ones** — `get_order_status(order_id)` instead of `run_sql(query)`; the schema itself becomes the permission boundary
- **Scoped credentials** — tools carry the *end user's* permissions, not a service account's; the agent can never read data its user couldn't
- **Read/write separation** — reads auto-approved, writes gated; separate tools so a read-only session physically lacks write capability
- **Parameter validation server-side** — treat every tool call like an untrusted API request ([[11.03 LLM Tool Calls]] args are model-generated); validate, allowlist, bound
- **No secrets in context** — tools resolve credentials internally; anything in the prompt can be exfiltrated

## Execution Isolation

| Layer | What it contains |
|---|---|
| **Process sandbox** (containers, gVisor, firejail) | Code execution, file system access |
| **Network egress allowlist** | Exfiltration channels — the third leg of the lethal trifecta ([[11.53 Prompt Injection]]) |
| **Filesystem scoping** | Workspace-only reads/writes; no `~/.ssh`, no env files |
| **Ephemeral environments** | Persistence of anything malicious across sessions |
| **Rate limits & budgets** | Runaway loops, mass actions (bulk email, bulk deletes) |

## Human-in-the-Loop Gates

Gate on **consequence, not confidence**:

- **Irreversible** (delete, send, pay, deploy) → explicit confirmation showing the *exact* action and arguments
- **Outward-facing** (email, posts, PRs) → human review; approval in one context doesn't extend to the next
- **Privilege escalation** (new scopes, new tools mid-session) → re-authorization
- Beware **confirmation fatigue** — if everything needs approval, users rubber-stamp; auto-approve the truly safe reads so the prompts that remain mean something

## Trust Boundaries in Multi-Tool Systems

> [!WARNING] Tool results are untrusted input
> Every tool that returns external content (web fetch, email read, RAG retrieval, [[11.08 MCP Server]] responses) injects third-party text into context. After the agent reads attacker-controllable content, downgrade its remaining session: restrict sensitive tools, or route via a quarantined model that has no tool access (dual-LLM pattern).
> With MCP specifically: a malicious or compromised server controls tool *descriptions* too — vet servers like you vet dependencies.

## Practical Use Cases

- Coding agents: sandboxed shell + egress allowlist + gated `git push`
- Email/calendar assistants: draft-only by default; send requires click-through
- Customer-support agents: per-customer scoped API tokens, refunds capped and logged
