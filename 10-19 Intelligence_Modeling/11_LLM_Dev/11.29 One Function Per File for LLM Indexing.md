# One Function Per File for LLM Indexing

## Overview
A radical extension of the SRP principle for **AI-assisted codebases**: structure code so that **each file contains exactly one function, class, or cohesive unit**. This maximizes [[11.23 Workspace Indexing]] precision, minimizes tokens pulled into the context window, and reduces [[11.28 Context Rot]] caused by irrelevant code competing for attention.

> [!INFO] The Core Insight
> AI assistants retrieve **whole files or large chunks** during context assembly. A 1500-line file forces the model to load 1500 lines of mostly-irrelevant code. A 30-line file loads only what matters.

---

## Why Smaller Files = Better LLM Performance

### 1. Retrieval Granularity
Workspace indexing chunks files for embedding. **Smaller files = more precise chunks = better semantic match.**

```
Monolithic file (800 lines, 12 functions)
└─ Embedded as ~8 chunks of mixed concerns
   → Retrieval returns chunks with 80% irrelevant code

One function per file (40 lines each)
└─ Each file = 1 focused chunk
   → Retrieval returns exactly the relevant function
```

### 2. Token Budget Efficiency

| File Strategy | Avg file size | Tokens per file pulled | Useful signal ratio |
|---|---|---|---|
| Monolithic (god module) | 1500 lines | ~6,000 tokens | ~10% |
| Grouped by domain | 400 lines | ~1,600 tokens | ~30% |
| **One function per file** | **40 lines** | **~160 tokens** | **~95%** |

The model can fit **30+ relevant functions** in the same budget as one monolith.

### 3. Cleaner Import Graph
The import statement *is* the dependency declaration. With one function per file, imports become a precise map of what code actually uses what — Copilot follows this map.

```python
# ❌ Monolith: unclear which helper is used where
from utils import (parse_date, format_currency, validate_email,
                   chunk_text, retry_with_backoff, ...)

# ✅ One per file: dependency is explicit and minimal
from utils.parse_date import parse_date
from utils.chunk_text import chunk_text
```

### 4. Reduced Context Rot
Per [[11.28 Context Rot]], distractor code in the context window degrades model accuracy. Single-purpose files **eliminate distractors** by construction.

## Practical Structure

### Directory Layout
```
src/
├── chunking/
│   ├── chunk_by_tokens.py        # 1 function
│   ├── chunk_by_sentence.py      # 1 function
│   ├── chunk_by_semantic.py      # 1 function
│   └── types.py                  # Shared types only
├── retrieval/
│   ├── embed_query.py
│   ├── search_vector_store.py
│   └── rerank_results.py
└── ...
```

That's the entire file. ~30 lines. Copilot loads it whole, with full context.

## When This Helps Most

| Scenario | Benefit |
|---|---|
| **Large codebases (>50K LOC)** | Indexing precision improves dramatically |
| **AI-first development workflows** | Copilot/Cursor suggestions become near-perfect |
| **RAG over your own code** | Each file = one retrievable unit of knowledge |
| **Microservices / serverless** | Aligns with one-function-per-handler patterns |
| **Onboarding new contributors** | New devs (and LLMs) can read one file end-to-end |

## When It Doesn't Help

| Scenario | Why |
|---|---|
| Tightly coupled internal helpers | Splitting forces artificial public APIs |
| Performance-critical hot paths | Import overhead and indirection cost real time |
| Small projects (<5K LOC) | Overhead exceeds benefit |
| Class methods with shared state | Class cohesion matters more than file size |

---

## Trade-offs

| Pro | Con |
|---|---|
| Maximum LLM context efficiency | More files to navigate manually |
| Minimal token waste | Risk of over-fragmentation |
| Crystal-clear dependency graph | Boilerplate (imports, docstrings) per file |
| Easy parallel work / no merge conflicts | Harder to see "the whole picture" without an MOC |
| Trivial unit testing (1 file = 1 test file) | Weaker IDE jump-to-definition payoff |

> [!WARNING] Don't fragment cohesive units
> A class with 5 methods that share private state belongs in **one file**. The "one function per file" rule is for **independent, composable** units — not for breaking apart cohesive abstractions.

---

## Hybrid Rule of Thumb

Don't be dogmatic. Use this tiered policy:

1. **One file per public function** when the function is independently useful
2. **One file per class** when the class is the cohesive unit
3. **Group only** when functions are private helpers used by exactly one caller (then inline or co-locate)
4. **Hard cap: 200 lines per file** — split anything larger

---

## Index-Friendly Naming

File names become **search keys** in the workspace index. Use the same name as the function:

```
✅ chunk_by_tokens.py        → contains chunk_by_tokens()
✅ embed_query.py            → contains embed_query()
❌ utils.py                  → contains 12 unrelated things
❌ helpers.py                → indexing nightmare
```

Copilot's symbol resolver and the embedding index both benefit when **filename = symbol name**.
