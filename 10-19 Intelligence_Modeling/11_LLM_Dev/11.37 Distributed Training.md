# 11.37 Distributed Training

## Overview
**Distributed training** splits an LLM training job across multiple GPUs (and nodes) to handle models and batches too large for one device. The main strategies — **data, tensor, pipeline, and sequence parallelism**, plus **ZeRO/FSDP sharding** — are composed into **3D / 4D parallelism** for frontier-scale training.

## Parallelism Strategies

| Strategy | What's Split | Communication | When to Use |
| --- | --- | --- | --- |
| **DDP (Data Parallel)** | Batch across GPUs; full model on each | All-reduce gradients | Model fits on one GPU |
| **ZeRO / FSDP** | Optimizer states → grads → params | All-gather + reduce-scatter | Model too big for one GPU |
| **Tensor Parallel (TP)** | Individual layer matmuls (column/row split) | All-reduce per layer (fast intra-node) | Single layer too big |
| **Pipeline Parallel (PP)** | Layers across GPUs (micro-batches in flight) | Point-to-point activations | Many layers, slow interconnect |
| **Sequence Parallel (SP)** | Sequence dimension (LayerNorm, dropout) | Small all-gather | Long contexts |
| **Expert Parallel (EP)** | MoE experts across GPUs | All-to-all routing | MoE models |

## ZeRO Stages (DeepSpeed)
> [!INFO] ZeRO Progressively Shards State
> ZeRO removes memory redundancy in DDP by sharding what each GPU holds.

| Stage | Sharded | Memory per GPU (vs DDP) | Comm Overhead |
| --- | --- | --- | --- |
| ZeRO-1 | Optimizer states | ~4× less | Same as DDP |
| ZeRO-2 | + Gradients | ~8× less | Same as DDP |
| ZeRO-3 | + Parameters | $N\times$ less ($N$ = world size) | ~1.5× DDP |

**FSDP** (PyTorch native) is functionally equivalent to **ZeRO-3** with similar API and performance.

## Tensor Parallelism (Megatron-style)
- **Column-parallel linear**: Split weight matrix by columns; each GPU computes a slice; all-gather output
- **Row-parallel linear**: Split by rows; all-reduce after matmul
- **MLP block**: Column-parallel up-proj + row-parallel down-proj → one all-reduce
- **Attention block**: Heads split across GPUs; one all-reduce
- **Constraint**: Heavy communication → keep TP **within a node** (NVLink), typically TP=2/4/8

## Pipeline Parallelism
- **Layers split into stages**, each on a different GPU
- **Micro-batches** flow through the pipeline to fill bubbles
- **Schedules**: GPipe (simple), 1F1B (interleaved, less memory), Zero-Bubble (newest)
- **Bubble overhead**: $\frac{P-1}{M}$ where $P$ = stages, $M$ = micro-batches

## 3D / 4D Parallelism
Frontier models combine all three (or four with sequence parallel):

```mermaid
graph LR
    A[Global Batch] --> B[Data Parallel<br/>across replicas]
    B --> C[Pipeline Parallel<br/>across nodes]
    C --> D[Tensor Parallel<br/>within node]
    D --> E[GPU]
```

Example: **GPT-3 175B training** used DP=64, PP=8, TP=8 = 4096 GPUs.

## FSDP vs DeepSpeed ZeRO
| Aspect | FSDP (PyTorch) | DeepSpeed ZeRO |
| --- | --- | --- |
| Native PyTorch | Yes | Plugin |
| ZeRO-3 equivalent | Yes | Yes |
| Tensor / Pipeline parallel | Via `torch.distributed` | Built-in |
| CPU offload | Yes | Yes (more options) |
| Mature ecosystem | Growing fast | Established |
| Recommended for | New projects, simplicity | Mature pipelines, MoE, ZeRO-Infinity |

## Practical Use Cases
- **Single-node multi-GPU (8×A100/H100)**: FSDP or ZeRO-3 + activation checkpointing
- **Multi-node SFT (7B–70B)**: FSDP + sequence parallel for long context
- **Frontier pre-training (100B+)**: Megatron-LM style 3D parallelism (TP+PP+DP)
- **MoE models**: Expert parallel + tensor parallel hybrid

> [!TIP] Choosing a Strategy
> 1. Model fits on one GPU → **DDP**
> 2. Model fits on one node (8 GPUs) → **FSDP / ZeRO-3**
> 3. Model fits on a few nodes → **FSDP + Tensor Parallel**
> 4. Frontier scale (100B+) → **3D parallelism** (Megatron-LM, NeMo)

> [!WARNING] Common Pitfalls
> - **TP across nodes**: Disastrous latency; keep TP within NVLink domain
> - **Too few micro-batches**: Pipeline bubble dominates; aim for $M \geq 4P$
> - **Gradient accumulation + ZeRO-3**: Triggers extra all-gathers; tune carefully
> - **CPU offload as a crutch**: Solves OOM but tanks throughput — prefer more GPUs
> - **Communication-bound jobs**: Profile with NSight / PyTorch Profiler before scaling further
