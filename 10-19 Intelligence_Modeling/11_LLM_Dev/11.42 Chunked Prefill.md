# 11.42 Chunked Prefill

## Overview
**Chunked prefill** splits a long prompt's prefill phase into **smaller token chunks** that are processed across multiple iterations, allowing the scheduler to **co-batch prefill chunks with ongoing decode steps**. This eliminates head-of-line blocking, smooths latency, and significantly improves throughput in modern LLM serving engines.

## The Problem It Solves
LLM inference has two phases with very different characteristics:

| Phase | Compute Pattern | Bottleneck | Latency Sensitivity |
| --- | --- | --- | --- |
| **Prefill** | One forward pass over $N$ prompt tokens | **Compute-bound** (FLOPs) | TTFT (time-to-first-token) |
| **Decode** | One token per step, autoregressive | **Memory-bound** (HBM bandwidth) | TPOT (time per output token) |

> [!WARNING] Head-of-Line Blocking
> A new request with a 32k-token prompt monopolizes the GPU for 100s of ms during prefill. All ongoing decode requests **stall**, causing TPOT spikes for every concurrent user.

## How Chunked Prefill Works

```mermaid
graph LR
    A[New 32k prompt] --> B[Split into chunks<br/>e.g., 512 tokens each]
    B --> C[Iteration 1:<br/>chunk 1 + decode batch]
    C --> D[Iteration 2:<br/>chunk 2 + decode batch]
    D --> E[...continues until prefill done]
    E --> F[Decode phase begins for new request]
```

- **Chunk size** (e.g., 512–2048 tokens) determines how many prefill tokens process per iteration
- Each iteration runs a **mixed batch**: one prefill chunk + many decode tokens from other requests
- Prefill is **compute-bound**, decode is **memory-bound** → they share the GPU well
- TTFT increases slightly for the long-prompt request, but TPOT for ongoing requests stays stable

## Benefits
- **Eliminates HOL blocking**: No single long prompt freezes the cluster
- **Higher throughput**: GPU stays busy with mixed compute + memory work (Sarathi-Serve reports 2–3×)
- **Smoother TPOT**: Tail latency for decode requests drops dramatically
- **Better SLA compliance**: Predictable per-token latency for streaming

## Trade-offs

| Metric | Without Chunked Prefill | With Chunked Prefill |
| --- | --- | --- |
| TTFT (long prompt) | Lower (single shot) | Slightly higher (split across chunks) |
| TPOT (other requests) | Spiky, can stall | Smooth, stable |
| Throughput | Lower | Higher |
| GPU utilization | Bursty | Steady |

> [!TIP] Tuning Chunk Size
> - **Small chunks (256–512)**: Best decode latency, slight prefill throughput loss
> - **Large chunks (2048+)**: Better prefill efficiency, more decode interference
> - **Sweet spot**: Match chunk size to where the workload transitions from compute- to memory-bound (often ~512–1024 on H100)

## Engine Support
- **vLLM**: `enable_chunked_prefill=True` (default in recent versions)
- **SGLang**: Built into the scheduler
- **TensorRT-LLM**: `enable_chunked_context`
- **DeepSpeed-MII / Sarathi-Serve**: Original research implementation

## Practical Use Cases
- **High-concurrency chat services** with mixed short and long prompts
- **RAG pipelines**: Long retrieved-context prefills don't block other users
- **Code assistants**: Large file context (10k+ tokens) without freezing other sessions
- **Multi-tenant inference**: Fairness across requests with vastly different prompt lengths

## Combining with Other Optimizations
- **+ [[11.39 Quantization]]**: Smaller weights → faster prefill chunks
- **+ [[11.41 Speculative Decoding]]**: Decode side gets speculative gains; prefill chunked separately
- **+ [[11.21 Prompt Caching]]**: Cached prefix skips prefill entirely; chunking handles the uncached tail
- **+ [[11.38 Flash Attention]]**: Required for efficient long-prompt prefill chunks

> [!INFO] Continuous Batching + Chunked Prefill
> Modern serving = **continuous batching** (per-iteration scheduling) + **chunked prefill** (split long prompts) + **paged KV cache** (memory efficiency). Together they form the standard high-throughput LLM serving stack pioneered by vLLM and Sarathi-Serve.

> [!WARNING] Common Pitfalls
> - **Tiny chunk size**: Overhead per iteration dominates; throughput drops
> - **No chunking for small prompts**: Always-chunk policies hurt short-prompt TTFT
> - **Ignoring Flash Attention compatibility**: Some attention backends don't support partial prefills cleanly
> - **Under-tuned mixed batch caps**: If max-tokens-per-iter is too low, decode requests starve
