# 11.40 Pruning & Sparsity

## Overview
**Pruning** removes weights (or whole structures) from a trained model to reduce size and accelerate inference. **Sparsity** is the resulting state — most weights are zero. For LLMs, **structured 2:4 sparsity** and **layer/depth pruning** are the most practical because hardware can actually exploit them.

## Pruning Taxonomy

| Type | What's Removed | Hardware Speedup | Quality Impact |
| --- | --- | --- | --- |
| **Unstructured** | Individual weights | Minimal (needs special kernels) | Lowest |
| **2:4 Structured** | 2 of every 4 weights = 0 | **2× on Ampere+ Tensor Cores** | Low |
| **Block / N:M** | Fixed sparsity pattern per block | Moderate | Low–Moderate |
| **Channel / Filter** | Whole rows/columns of weight matrices | Real (smaller matmuls) | Moderate |
| **Layer / Depth** | Entire transformer layers | Real (fewer layers) | Moderate–High |
| **Head pruning** | Attention heads | Real (smaller attention) | Low–Moderate |

## Key Concepts
- **Magnitude pruning**: Drop weights with smallest absolute value
- **Importance scores**: Use gradient × weight (Taylor), Hessian-based (SparseGPT, Wanda)
- **One-shot vs iterative**: Modern LLM pruning is mostly **one-shot** (no retraining)
- **Hardware-aware**: 2:4 sparsity is the only pattern with **native GPU acceleration**

## Methods for LLMs

| Method | Pattern | Calibration | Quality at 50% Sparsity |
| --- | --- | --- | --- |
| **Magnitude** | Unstructured / 2:4 | None | Poor |
| **SparseGPT** | Unstructured / 2:4 | Yes | Good — near baseline |
| **Wanda** | Unstructured / 2:4 | Yes (forward only) | Good — simpler than SparseGPT |
| **LLM-Pruner** | Structured (channels) | Yes + small fine-tune | Moderate |
| **ShortGPT / LayerDrop** | Whole layers | Yes | Surprisingly good (many layers redundant) |

## 2:4 Structured Sparsity
> [!INFO] The Only Free Lunch
> NVIDIA Ampere/Hopper Tensor Cores natively support **2:4 sparsity**: in every group of 4 consecutive weights, exactly 2 must be zero. Hardware skips the zeros for **2× matmul throughput**.

- Apply via SparseGPT or Wanda with the `2:4` constraint
- Combine with [[11.39 Quantization|INT4 quantization]] → **~4× total compression** with minor quality loss
- Supported in TensorRT-LLM, cuSPARSELt

## Layer / Depth Pruning
> [!TIP] Many LLM Layers Are Redundant
> Studies (ShortGPT, "The Unreasonable Ineffectiveness of the Deeper Layers") show **20–40% of middle layers** can be removed from large LLMs (Llama-2 70B, Mistral) with minimal quality loss after light healing.

- **Block influence metric**: Measure cosine similarity between layer input and output; low → prunable
- **Healing**: Short LoRA fine-tune on a small dataset to recover quality
- **Best targets**: Middle layers in deep models; first and last layers are critical
- **Result**: A 70B → 50B "pruned" model that runs ~30% faster

## Sparsity vs Quantization
| Dimension | Quantization | Pruning / Sparsity |
| --- | --- | --- |
| Size reduction | 2–8× (precision) | 2× (2:4) or more (structured) |
| Hardware support | Excellent (INT4/8, FP8) | Limited (only 2:4) |
| Quality at aggressive levels | Good (INT4) | Worse (>50% unstructured) |
| Combine with the other? | **Yes** — common to do both | Yes |
| Maturity for LLMs | Production standard | Research / niche |

## Practical Use Cases
- **Edge inference**: Combine layer pruning + INT4 quantization for laptop/phone deployment
- **Latency-critical serving**: 2:4 sparsity + Tensor Cores for 2× attention/MLP speedup
- **Model distillation prep**: Pruned model as starting point for further compression
- **Research / cost reduction**: Trim known-redundant layers from open-weight models

> [!WARNING] Common Pitfalls
> - **Unstructured sparsity ≠ speedup**: Without specialized kernels, sparse matmul is **slower** than dense
> - **Pruning destroys reasoning**: Math, code, and CoT quality drops faster than chat quality
> - **Compounding losses**: Prune + quantize + distill stacks errors — evaluate at every stage
> - **Sparsity ratios on log loss look fine but break downstream tasks** — always run task evals
> - **Healing requires care**: LoRA fine-tune after pruning needs a representative dataset, not just instruction data
