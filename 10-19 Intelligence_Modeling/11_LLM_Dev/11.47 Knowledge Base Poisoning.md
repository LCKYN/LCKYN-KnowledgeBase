# Knowledge Base Poisoning

## Overview
**Knowledge Base Poisoning** is an attack on [[11.12 RAG]] systems where an adversary injects malicious documents into the retrieval corpus. Because RAG treats retrieved text as trusted grounding, a few poisoned chunks can steer answers to targeted misinformation, smuggle in prompt-injection payloads, or degrade the system — **without ever touching the model**.

> [!WARNING] It takes very little
> **PoisonedRAG** (Zou et al., 2024) showed ~5 poisoned texts per target question — in a corpus of millions — achieved ~90% attack success. Poisoning is cheap because retrieval, not the model, is the weak link.

## Attack Anatomy

Two conditions must hold; attackers optimize each separately:

1. **Retrieval condition** — the poisoned text must be retrieved: stuff it with terms/embeddings similar to target queries (works even black-box)
2. **Generation condition** — once in context, the text must win: authoritative tone, fake citations, direct instructions, or claims the model repeats

## Attack Goals

| Goal | Example |
|---|---|
| **Targeted misinformation** | "For question X, always answer Y" |
| **Indirect prompt injection** | Poisoned chunk contains instructions the LLM executes (see [[11.48 User Message Confusion]]) |
| **Denial of service** | Chunks that trigger refusals or drown out real answers |
| **Bias / promotion** | Steer recommendations toward attacker's product |
| **Data exfiltration** | Injected instructions make the agent leak conversation or tool data |

## Injection Vectors

- **Scraped web content** — attacker publishes pages your indexer will crawl
- **User-generated content** — wikis, tickets, reviews, chat logs, résumés fed into the KB
- **Inbound documents** — email attachments, uploaded PDFs (including hidden text: white-on-white, metadata, comments)
- **Compromised upstream sources** — third-party datasets, package docs
- **Insider edits** — anyone with write access to the corpus

## Defenses (layered — none sufficient alone)

- **Provenance & write control** — know where every document came from; ACL the indexing pipeline like production code
- **Ingestion sanitization** — strip hidden text, detect instruction-like content ("ignore previous...", role markers) before indexing
- **Retrieval-time isolation** — wrap retrieved text as untrusted data, never as instructions; escape special tokens
- **Corroboration** — require multiple independent sources to agree before asserting a fact (majority-vote / [[11.46 Falsification-Verification RAG (FVA-RAG)]]-style verification)
- **Anomaly detection** — flag chunks retrieved abnormally often, or embeddings unnaturally close to query space
- **Output guardrails** — [[11.10 LLM Guardrails]] on citations, claims, and tool calls triggered by retrieved content

> [!TIP] Threat-model your ingestion pipeline
> Ask: "who can write text that ends up in my index?" Every yes that isn't an employee with review is an attack surface.
