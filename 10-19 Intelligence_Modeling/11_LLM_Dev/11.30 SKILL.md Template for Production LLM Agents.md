# SKILL.md Template for Production LLM Agents

## Overview
A **SKILL.md** is a structured instruction file that an LLM coding assistant (Claude, Copilot, Cursor) loads on-demand to perform a specialized task with **tested, opinionated best practices**. Think of it as **prompt engineering as a versioned artifact** — a reusable, domain-specific system prompt that lives in the repo.

> [!INFO] Why SKILL.md beats ad-hoc prompts
> - **Versioned** in git, reviewed in PRs
> - **Composable** — multiple skills load for cross-cutting tasks
> - **Discoverable** by the agent itself (it scans descriptions and loads what fits)
> - **Token-efficient** — only loaded when relevant, not always in context

---

## Anatomy of a Good SKILL.md

| Section | Purpose | Length |
|---|---|---|
| **Frontmatter / Header** | Name, description, when-to-use trigger | 5–10 lines |
| **Trigger conditions** | Concrete examples of user requests that should activate this skill | 5–15 lines |
| **Core principles** | The 3–7 non-negotiable rules | 10–30 lines |
| **Step-by-step workflow** | The exact procedure to follow | 30–80 lines |
| **Code templates** | Minimal, copy-pasteable scaffolds | 50–150 lines |
| **Anti-patterns** | What NOT to do, with reasons | 10–30 lines |
| **Verification checklist** | How the agent confirms it's done | 10–20 lines |

**Total target: 200–400 lines.** Longer skills get truncated or ignored.

# Production LLM Agent (Python)

## When to Use This Skill
Activate when the user requests:
- "Build an LLM agent that..."
- "Create an AI assistant that calls tools..."
- "Make an autonomous agent for [task]..."
- "Scaffold a Python agent using OpenAI/Anthropic/etc."

Do NOT use for: simple one-shot completions, RAG-only pipelines (use `rag-pipeline` skill), or chatbot UIs (use `chatbot-ui` skill).

---

## Core Principles (Non-Negotiable)

1. **Typed I/O at every boundary** — Pydantic models for tool args, tool results, agent state
2. **One tool = one file = one function** — see One Function Per File pattern
3. **Structured output via JSON schema or constrained generation** — never parse free-text
4. **Bounded loops** — every agent loop has a max-iteration cap
5. **Retries with exponential backoff** on transient errors only (rate limits, 5xx)
6. **Observability first** — log every LLM call, tool call, and decision with a trace ID
7. **Costs are tracked** — token counts and $ per request emitted as metrics
8. **Tests use recorded fixtures** — no live LLM calls in CI

## Step-by-Step Workflow

### Step 1: Define Pydantic models first
Create `models.py` with:
- `AgentState` — message history, iteration count, total cost
- `ToolCall` and `ToolResult` envelopes
- One input/output model per tool

### Step 2: Build the LLM client wrapper
In `client.py`, wrap the SDK with:
- `tenacity` retries (only on `RateLimitError`, `APIConnectionError`, 5xx)
- Cost tracking (read `usage` from response, multiply by per-token price)
- Structured logging with trace_id

### Step 3: Implement tools (one per file)
Each tool file exports:
- A Pydantic args model
- A Pydantic result model
- The function itself with full type annotations and docstring

### Step 4: Build the bounded agent loop
In `agent.py`:
- Accept `max_iterations: int = 10`
- Each iteration: call LLM → parse tool calls → execute → append results
- Halt on: final answer, max iterations, or unrecoverable error
- Return full `AgentState` (including history) for inspection

### Step 5: Tests with recorded fixtures
Use `pytest-recording` (VCR) or hand-crafted fixtures:
- Test each tool in isolation with mocked external deps
- Test the loop with a recorded LLM transcript
- Assert on: final answer, tool call sequence, total cost, iteration count

### Step 6: Observability
- `structlog` for JSON logs with trace_id
- Emit metrics: `agent.iterations`, `agent.cost_usd`, `tool.duration_ms`, `tool.failures`
- Add OpenTelemetry spans around LLM and tool calls

## Code Templates

### `models.py`
```python
from pydantic import BaseModel, Field
from typing import Literal

class Message(BaseModel):
    role: Literal["system", "user", "assistant", "tool"]
    content: str
    tool_call_id: str | None = None

class ToolCall(BaseModel):
    id: str
    name: str
    args: dict

class AgentState(BaseModel):
    messages: list[Message]
    iteration: int = 0
    total_cost_usd: float = 0.0
    finished: bool = False
    final_answer: str | None = None
```

### `client.py` (skeleton)
```python
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type
from openai import OpenAI, RateLimitError, APIConnectionError
import structlog

log = structlog.get_logger()

class LLMClient:
    def __init__(self, model: str, settings: Settings):
        self._client = OpenAI(api_key=settings.openai_api_key)
        self.model = model

    @retry(
        stop=stop_after_attempt(3),
        wait=wait_exponential(multiplier=1, min=2, max=10),
        retry=retry_if_exception_type((RateLimitError, APIConnectionError)),
    )
    def complete(self, messages: list[dict], tools: list[dict], trace_id: str):
        log.info("llm.call.start", trace_id=trace_id, model=self.model)
        resp = self._client.chat.completions.create(
            model=self.model, messages=messages, tools=tools
        )
        cost = self._calculate_cost(resp.usage)
        log.info("llm.call.done", trace_id=trace_id, cost_usd=cost)
        return resp, cost
```

### `tools/search_web.py` (one tool per file)
```python
from pydantic import BaseModel, Field

class SearchWebArgs(BaseModel):
    query: str = Field(..., description="Search query, max 10 words")
    top_k: int = Field(5, ge=1, le=20)

class SearchWebResult(BaseModel):
    results: list[dict]

def search_web(args: SearchWebArgs) -> SearchWebResult:
    """Search the public web and return top-k results with title, url, snippet."""
    # impl
    return SearchWebResult(results=[...])

TOOL_SPEC = {
    "type": "function",
    "function": {
        "name": "search_web",
        "description": search_web.__doc__,
        "parameters": SearchWebArgs.model_json_schema(),
    },
}
```

### `agent.py` (bounded loop)
```python
def run_agent(user_input: str, max_iterations: int = 10) -> AgentState:
    state = AgentState(messages=[
        Message(role="system", content=load_system_prompt()),
        Message(role="user", content=user_input),
    ])
    trace_id = uuid4().hex

    while state.iteration < max_iterations and not state.finished:
        state.iteration += 1
        resp, cost = client.complete(
            messages=[m.model_dump() for m in state.messages],
            tools=ALL_TOOL_SPECS,
            trace_id=trace_id,
        )
        state.total_cost_usd += cost

        tool_calls = parse_tool_calls(resp)
        if not tool_calls:
            state.final_answer = resp.choices[0].message.content
            state.finished = True
            break

        for call in tool_calls:
            result = TOOL_REGISTRY[call.name](call.args)
            state.messages.append(Message(
                role="tool", tool_call_id=call.id, content=result.model_dump_json()
            ))

    return state
```

## Anti-Patterns

| Don't | Why | Do Instead |
|---|---|---|
| Unbounded `while True` loops | Runaway cost / infinite recursion | `max_iterations` cap |
| Parse free-text tool calls with regex | Fragile, breaks on model updates | JSON schema / native tool calling |
| Retry on every exception | Masks real bugs, wastes money | Whitelist transient errors only |
| Log full message history at INFO | Leaks PII, blows up log volume | DEBUG level + redaction |
| Hardcode API keys / model names | Breaks across envs | `pydantic-settings` from env |
| Live LLM calls in unit tests | Flaky, expensive, slow CI | VCR cassettes / fixtures |
| Single 800-line `agent.py` | LLM and humans both lose context | One tool per file, slim loop |
| Ignore token usage | Surprise $10K bills | Track cost per request, alert on threshold |

## Verification Checklist

Before declaring done, confirm:
- [ ] All tools have Pydantic args + result models
- [ ] Each tool lives in its own file with matching filename
- [ ] Agent loop has `max_iterations` parameter (default ≤ 10)
- [ ] LLM client retries only on transient errors
- [ ] Cost is tracked and returned in `AgentState`
- [ ] All LLM calls and tool calls are logged with `trace_id`
- [ ] Tests run without network access (fixtures/VCR)
- [ ] Config uses `pydantic-settings`, no hardcoded secrets
- [ ] System prompt lives in `prompts/system.md`, not inline string
- [ ] `pyproject.toml` pins major versions of `openai`/`anthropic`, `pydantic>=2`, `tenacity`, `structlog`

## Skill Description Best Practices

The `description` field is what triggers loading. Make it:

- **Specific about scope** — "production-ready Python LLM agent with tools" not "AI stuff"
- **Include trigger keywords** — list the user phrases that should activate it
- **State exclusions** — what this skill is NOT for
- **Under 200 chars** — long descriptions get truncated in skill discovery

✅ Good:
> "Build a production-ready LLM agent in Python with tool calling, structured output, retries, observability, and tests. Use when the user asks to create, scaffold, or build an LLM agent, AI assistant, autonomous agent, or tool-using LLM in Python."

❌ Bad:
> "Helps with AI agents."
