# Context Rot

## Overview
**Context Rot** is the phenomenon where an LLM's performance **degrades non-uniformly as input length grows**, even when staying well within the advertised context window. A model rated for 1M tokens does **not** maintain its short-context quality across that whole range — accuracy, recall, and reasoning all decay as context fills up.

> [!INFO] Source
> Coined and quantified by **Chroma Research** (2024) in their report *"Context Rot: How Increasing Input Tokens Impacts LLM Performance"*, evaluating 18 frontier models (GPT-4.1, Claude 4, Gemini 2.5, Qwen3, etc.).

---

## Key Findings

| Finding | Implication |
|---|---|
| Performance is **non-uniform** across input length | "128K context" ≠ usable at 128K |
| **Needle-in-haystack (NIAH)** tests are too easy — they overstate long-context ability | Real tasks require semantic matching, not lexical |
| **Distractors** (semantically similar but wrong answers) sharply degrade recall as context grows | RAG reranking matters more at scale |
| **Needle-question similarity** dictates difficulty more than raw token count | Low-similarity retrieval fails first |
| **Haystack structure matters** — shuffled context often outperforms coherent context | Models exploit positional/lexical shortcuts |
| Models fail differently: some hallucinate, some refuse, some over-hedge | No single mitigation works universally |

---

## Why It Happens

- **Attention dilution** — softmax distributes attention across more tokens, weakening signal on relevant ones (related: [[11.01 Attention Mechanism]])
- **Position bias** — "lost in the middle"; tokens at start/end recalled better than middle
- **Training distribution mismatch** — most training sequences are far shorter than max context
- **Distractor interference** — semantically related noise competes with the true answer

---

## Practical Mitigations

### 1. Treat Context as a Scarce Resource
Don't dump everything in. Apply [[11.16 Context Window Management]] aggressively — the cheapest token is the one you don't send.

### 2. Aggressive Reranking in RAG
Distractors are the main killer. Use a strong cross-encoder reranker and **trim to top-k=3–10**, not top-k=50. See [[11.12 RAG]].

### 3. Front-Load and Tail-Load Critical Info
Place the most important instructions/data at the **beginning or end** of the prompt — middle positions degrade fastest.

### 4. Chunk-and-Verify Pipelines
For long documents, run multiple short-context calls + aggregation rather than one mega-context call.

### 5. Measure on Your Task, Not NIAH
Build evals that mirror real distractor density and semantic difficulty (see [[11.15 LLM Evaluation Metrics]]).

### 6. Cache the Stable Prefix
If long context is unavoidable, use [[11.21 Prompt Caching]] to amortize cost — but caching does **not** fix accuracy decay.

---

## Context Rot vs. Related Concepts

| Concept | What it describes |
|---|---|
| **Context Rot** | Quality degradation as input length grows |
| **Context window limit** | Hard token ceiling (truncation) |
| **Lost in the middle** | Specific positional failure mode (subset of rot) |
| **Hallucination** | Fabrication; rot increases its likelihood |

---

> [!WARNING] The "Big Context Window" Trap
> Marketing claims of 1M–10M token windows imply you can stuff entire codebases or document libraries in. **You can't — not effectively.** Build your pipeline as if the *useful* context is 32K–64K, regardless of the stated maximum.

> [!TIP] Rule of Thumb
> If your prompt exceeds ~50% of the model's stated window, you are likely losing measurable accuracy. Reduce, retrieve, or split.
